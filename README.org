* Introduction

This guide documents how to set up Emacs as a modern development environment, built entirely on packages that are either built-in or require minimal external dependencies. The approach favours Emacs's native infrastructure --- eglot over lsp-mode, flymake over flycheck, corfu over company --- and emphasises portability, including air-gapped and restricted environments.

Three concrete worked examples are covered in detail:

- *Java* (Linux): eglot + JDTLS + dape with java-debug-adapter
- *C#/.NET* (Windows): eglot + csharp-ls + dape with netcoredbg
- *Ada* (Linux/Windows): old ada-mode + eglot + gud/gdb

The principles and patterns apply equally to any language with LSP and DAP support (Python, Go, Rust, C/C++, etc.).

** What We Are Building

| Layer              | Tool          | Status in Emacs 30 | Role                                            |
|--------------------+---------------+--------------------+-------------------------------------------------|
| Language Server    | *eglot*       | Built-in (29+)     | LSP client: completion, diagnostics, navigation |
| Diagnostics        | *flymake*     | Built-in           | Displays errors/warnings inline                 |
| Documentation      | *eldoc*       | Built-in           | Shows documentation at point                    |
| Completion UI      | *corfu*       | External (MELPA)   | Popup completion interface                      |
| Debugging          | *dape*        | External (MELPA)   | Debug Adapter Protocol client                   |
| Project management | *project.el*  | Built-in           | Project root detection, file switching          |
| Xref navigation    | *xref*        | Built-in           | Go-to-definition, find-references               |
| Package management | *use-package* | Built-in (29+)     | Declarative package configuration               |

All of the "External" packages are single-file or small packages with no dependencies beyond Emacs itself.

* Part 1: Emacs Foundations

** Installing Emacs

Emacs 30.1 (or later) is recommended. It includes eglot, use-package, project.el, and flymake out of the box.

*** Linux (Package Manager)

#+begin_src bash
# Arch Linux
sudo pacman -S emacs

# Fedora
sudo dnf install emacs

# Ubuntu/Debian (may need a PPA for 30+)
sudo apt install emacs
#+end_src

*** Linux (From Source)

If your distribution ships an older Emacs (e.g. openSUSE Leap 15.4 ships 27.2), build from source:

#+begin_src bash
mkdir -p ~/emacs-builds && cd ~/emacs-builds
tar xzf emacs-30.1.tar.gz
cd emacs-30.1
./configure --prefix=$HOME/emacs-versions/emacs-30.1 \
    --with-x-toolkit=gtk3 --with-gnutls --with-xml2 \
    --with-jpeg --with-png --with-gif --with-tiff \
    --with-xpm --with-cairo --with-harfbuzz --with-rsvg
make -j$(nproc)
make install
#+end_src

Then add to PATH:
#+begin_src bash
echo 'alias emacs="$HOME/emacs-versions/emacs-30.1/bin/emacs"' >> ~/.bashrc
#+end_src

*** Windows

Download from the [[https://www.gnu.org/software/emacs/download.html][GNU Emacs download page]]. Install to a non-system location (e.g. =D:\source\emacs-30.1=) to avoid permission issues and keep everything portable.

*** macOS

#+begin_src bash
brew install emacs-plus@30 --with-native-comp
#+end_src

** The Built-in Ecosystem

One of the strengths of modern Emacs is how much works out of the box. Before reaching for external packages, understand what's already there:

- *eglot* --- LSP client that integrates with flymake, eldoc, xref, completion-at-point, and imenu. One package wires up six features.
- *flymake* --- Real-time diagnostic display. Eglot feeds it errors and warnings from the language server automatically.
- *eldoc* --- Shows documentation in the echo area (or a dedicated buffer). Eglot provides hover information through it.
- *xref* --- =M-.= (go to definition), =M-?= (find references), =M-,= (go back). Eglot provides the backend.
- *project.el* --- =C-x p f= (find file in project), =C-x p g= (grep project), =C-x p c= (compile). Detects project roots via version control.
- *completion-at-point* --- =C-M-i= or =M-TAB=. The raw completion interface that eglot populates. Corfu makes this prettier.
- *imenu* --- =M-x imenu= to jump to definitions in the current buffer. Eglot enhances it with LSP symbol information.

** Installing External Packages

For the packages not built into Emacs (corfu, dape, etc.), there are two approaches.

*** Online: From MELPA

#+begin_src elisp
;; Add MELPA to package archives (add to init.el)
(require 'package)
(add-to-list 'package-archives '("melpa" . "https://melpa.org/packages/") t)
(package-initialize)
#+end_src

Then install via =M-x package-install= or via =use-package= with =:ensure t=.

*** Offline: Manual Installation

For air-gapped environments, download package source files on a connected machine and copy them across.

#+begin_src bash
# Example: dape is a single file, no dependencies beyond Emacs 29+
git clone https://github.com/svaante/dape.git /tmp/dape
cp /tmp/dape/dape.el ~/.emacs.d/site-lisp/

# Example: corfu
git clone https://github.com/minad/corfu.git /tmp/corfu
cp -r /tmp/corfu ~/.emacs.d/site-lisp/corfu
#+end_src

Then load from =site-lisp= in your init:
#+begin_src elisp
(add-to-list 'load-path "~/.emacs.d/site-lisp")
(add-to-list 'load-path "~/.emacs.d/site-lisp/corfu")
#+end_src

* Part 2: Eglot (LSP Client)

Eglot is the built-in LSP client. It automatically configures flymake, eldoc, xref, and completion-at-point when connected to a language server.

** Basic Configuration

#+begin_src elisp
(use-package eglot
  :hook
  ((java-mode . eglot-ensure)
   (csharp-mode . eglot-ensure)
   ;; Add other language modes as needed:
   ;; (python-mode . eglot-ensure)
   ;; (rust-mode . eglot-ensure)
   ;; (go-mode . eglot-ensure)
   ))
#+end_src

=eglot-ensure= starts the language server automatically when you open a file in that mode. For manual control, use =M-x eglot= instead.

** Configuring Language Servers

Eglot knows about many language servers out of the box. For servers not in the default list, or to override the defaults, use =eglot-server-programs=.

*** Java: JDTLS (Eclipse JDT Language Server)

JDTLS is the standard Java language server. It supports Gradle and Maven projects, code completion, diagnostics, refactoring, and debugging (via a plugin).

**** Installing JDTLS

Download from [[https://download.eclipse.org/jdtls/milestones/][Eclipse JDTLS releases]] or build from source. The =bin/jdtls= script is a Python launcher that auto-detects the correct configuration.

For offline environments, extract the full JDTLS distribution to a known path:
#+begin_src bash
mkdir -p ~/.emacs.d/bin
tar xzf jdtls-<version>.tar.gz -C ~/.emacs.d/bin/jdtls
#+end_src

**** Eglot Configuration for JDTLS

#+begin_src elisp
(add-to-list 'eglot-server-programs
             `((java-mode java-ts-mode) .
               ("~/.emacs.d/bin/jdtls/bin/jdtls"
                :initializationOptions
                (:bundles ["~/.emacs.d/bin/jdtls/com.microsoft.java.debug.plugin-0.53.2.jar"]))))
#+end_src

The =:bundles= entry loads the java-debug-adapter plugin, which is needed for debugging (see the Dape section below).

**** JDTLS Memory Tuning

JDTLS can be memory-intensive on large projects. Increase the heap:
#+begin_src elisp
(add-to-list 'eglot-server-programs
             `((java-mode java-ts-mode) .
               ("~/.emacs.d/bin/jdtls/bin/jdtls"
                "--jvm-arg=-Xmx4G"
                :initializationOptions
                (:bundles ["~/.emacs.d/bin/jdtls/com.microsoft.java.debug.plugin-0.53.2.jar"]))))
#+end_src

Or edit =~/.emacs.d/bin/jdtls/config_linux/config.ini= directly.

*** C#: csharp-ls

** Performance Tuning: Disabling Server Capabilities

If eglot or the language server is sluggish, disable capabilities you don't need. This is especially useful on resource-constrained systems.

#+begin_src elisp
(setq eglot-ignored-server-capabilities
      '(
        ;; :hoverProvider                    ; Documentation on hover (eldoc)
        ;; :completionProvider               ; Code completion
        ;; :signatureHelpProvider            ; Function signature help
        ;; :definitionProvider               ; Go to definition (M-.)
        :typeDefinitionProvider           ; Go to type definition
        :implementationProvider           ; Go to implementation
        :declarationProvider              ; Go to declaration
        ;; :referencesProvider               ; Find references (M-?)
        ;; :documentHighlightProvider        ; Highlight symbol at point
        :documentSymbolProvider           ; List symbols in buffer (imenu)
        :workspaceSymbolProvider          ; List symbols in workspace
        :codeActionProvider               ; Code actions (quick fixes)
        :codeLensProvider                 ; Code lens
        :documentFormattingProvider       ; Format buffer
        :documentRangeFormattingProvider  ; Format region
        :documentOnTypeFormattingProvider ; On-type formatting
        ;; :renameProvider                   ; Rename symbol
        :documentLinkProvider             ; Highlight links
        :colorProvider                    ; Colour references
        :foldingRangeProvider             ; Code folding
        :executeCommandProvider           ; Custom server commands
        :inlayHintProvider                ; Inlay hints
        ))
#+end_src

Uncomment a line to re-enable that capability. The commented-out entries above represent a good baseline --- hover, completion, go-to-definition, references, highlight, and rename are the most commonly used features.

** Process Output Tuning

For large projects, increase the read buffer size to avoid bottlenecks:

#+begin_src elisp
(setq read-process-output-max (* 1024 1024))  ; 1MB (default is 4KB)
#+end_src

* Part 3: Flymake (Diagnostics)

Flymake is Emacs's built-in on-the-fly syntax checking framework. When eglot is active, it automatically feeds diagnostics from the language server into flymake.

** Configuration

#+begin_src elisp
(use-package flymake
  :hook (prog-mode . flymake-mode)
  :bind
  (("M-n" . flymake-goto-next-error)
   ("M-p" . flymake-goto-prev-error))
  :custom
  ;; Show diagnostics at end of line (Emacs 30+)
  ;; Set to nil if you find it too noisy
  (flymake-show-diagnostics-at-end-of-line t))
#+end_src

** Key Commands

| Key       | Command                            | Action                          |
|-----------+------------------------------------+---------------------------------|
| =M-n=     | =flymake-goto-next-error=          | Jump to next diagnostic         |
| =M-p=     | =flymake-goto-prev-error=          | Jump to previous diagnostic     |
| =C-c ! l= | =flymake-show-buffer-diagnostics=  | List all diagnostics in buffer  |
| =C-c ! p= | =flymake-show-project-diagnostics= | List diagnostics across project |

Diagnostics appear as underlined text with faces indicating severity (error, warning, note).

  )

* Part 5: Corfu (Completion UI)

Corfu provides a modern popup completion interface that works with Emacs's native =completion-at-point= system. Since eglot provides a completion-at-point backend, corfu automatically shows LSP completions.

** Configuration

#+begin_src elisp
(use-package corfu
  ;; :ensure t  ; or :load-path for offline
  :custom
  (corfu-auto nil)          ; No automatic popup (invoke with C-M-i)
  (corfu-cycle t)           ; Cycle through candidates
  (corfu-preselect 'first)  ; Pre-select first candidate
  (corfu-quit-no-match nil) ; Don't quit when no match
  :init
  (global-corfu-mode))
#+end_src

*** Auto-completion vs Manual Completion

Setting =corfu-auto= to =nil= means completion only appears when you invoke it (=C-M-i= or =M-TAB=). This avoids constant LSP requests which can cause slowdowns, especially on large projects or resource-constrained systems.

If you prefer auto-completion:
#+begin_src elisp
(setq corfu-auto t)
(setq corfu-auto-delay 0.2)   ; Delay in seconds
(setq corfu-auto-prefix 2)    ; Minimum characters before popup
#+end_src

** Corfu with Eglot

No additional configuration is needed. Eglot registers itself as a =completion-at-point-functions= backend, and corfu displays whatever that system provides.

To invoke completion manually: =C-M-i= (=completion-at-point=) or =M-TAB=.

** Java Debugging with JDTLS

Java debugging requires the *java-debug-adapter* plugin loaded as a JDTLS bundle (configured in the eglot section above). The debug adapter runs inside JDTLS itself, so dape needs a helper function to ask JDTLS to start a debug session.

*** Building java-debug-adapter

#+begin_src bash
git clone https://github.com/microsoft/java-debug.git
cd java-debug
./mvnw clean install -DskipTests
# Output: com.microsoft.java.debug.plugin/target/com.microsoft.java.debug.plugin-<VERSION>.jar
#+end_src

Copy the jar to your JDTLS directory and reference it in the =:bundles= option of your eglot configuration.

*** JDTLS Debug Session Helper

#+begin_src elisp
(defun dape--jdtls-start-debug-session (config)
  "Resolve classpath via JDTLS, start a debug session, and configure dape."
  (let ((server (eglot-current-server)))
    (unless server
      (user-error "Eglot is not active, run M-x eglot first"))
    (let* ((main-class (plist-get config :mainClass))
           (project-name (plist-get config :projectName))
           ;; Ask JDTLS to resolve the classpath
           (classpath-result (eglot-execute-command
                              server "vscode.java.resolveClasspath"
                              (vector main-class project-name)))
           (module-paths (aref classpath-result 0))
           (class-paths (aref classpath-result 1))
           ;; Start the debug adapter session inside JDTLS
           (port (eglot-execute-command
                  server "vscode.java.startDebugSession" nil)))
      (setq config (plist-put config 'port port))
      (setq config (plist-put config 'host "localhost"))
      (setq config (plist-put config :modulePaths module-paths))
      (setq config (plist-put config :classPaths class-paths))
      config)))
#+end_src

*** Java Launch Configurations

#+begin_src elisp
;; Launch a Java application
(add-to-list 'dape-configs
  '(java-launch
    modes (java-mode java-ts-mode)
    fn dape--jdtls-start-debug-session
    :type "java"
    :request "launch"
    :mainClass "com.example.Main"     ; <-- your main class
    :projectName "my-project"         ; <-- Gradle/Maven module name
    :cwd "/path/to/project"
    :console "internalConsole"
    :stopAtEntry t))

;; Attach to a running JVM (e.g. started with --debug-jvm)
(add-to-list 'dape-configs
  '(java-attach
    modes (java-mode java-ts-mode)
    host "localhost" port 5005
    :type "java"
    :request "attach"
    :hostName "localhost"
    :port 5005))
#+end_src

**** Usage

*Launch mode*: Open a Java file, ensure eglot is connected, then =F5= (=M-x dape=) and select a launch configuration.

*Attach mode*: Start the application with debug flags first:
#+begin_src bash
./gradlew myTask --debug-jvm  # Suspends on port 5005
#+end_src
Then =F5= and select the attach configuration.

** C#/.NET Debugging with netcoredbg

netcoredbg is Samsung's open-source .NET debugger that implements the DAP protocol.

*** Installing netcoredbg

Download from [[https://github.com/Samsung/netcoredbg/releases][Samsung's GitHub releases]] and extract to a known location.

*** C#/.NET Launch Configuration

#+begin_src elisp
(add-to-list 'dape-configs
  '(netcoredbg-launch
    modes (csharp-mode csharp-ts-mode)
    command "/path/to/netcoredbg"
    command-args ("--interpreter=vscode")
    :type "coreclr"
    :request "launch"
    :program "/path/to/bin/Debug/net9.0/MyApp.dll"
    :cwd "/path/to/project"
    :console "internalConsole"
    :stopAtEntry t))
#+end_src

*** Helper for Multiple .NET Projects

When working with a solution containing multiple projects, a helper function avoids repetition:

#+begin_src elisp
(defvar my-netcoredbg-path "/path/to/netcoredbg"
  "Path to netcoredbg executable.")
(defvar my-project-root "/path/to/solution"
  "Root directory of the solution.")
(defvar my-build-config "Debug"
  "Build configuration (Debug or Release).")

(defun my-dape-dotnet-config (component-name dll-name &optional stop-at-entry)
  "Create a dape configuration for a .NET component.
COMPONENT-NAME is the project directory name.
DLL-NAME is the DLL filename without extension."
  (let* ((component-dir (format "%s/%s" my-project-root component-name))
         (bin-path (format "%s/bin/%s/net9.0" component-dir my-build-config))
         (dll-path (format "%s/%s.dll" bin-path dll-name))
         (config-name (intern (format "netcoredbg-launch-%s"
                                      (downcase component-name)))))
    `(,config-name
      modes (csharp-mode csharp-ts-mode)
      command ,my-netcoredbg-path
      command-args ("--interpreter=vscode")
      :type "coreclr"
      :request "launch"
      :program ,dll-path
      :cwd ,component-dir
      :console "internalConsole"
      :stopAtEntry ,(if stop-at-entry t :json-false))))

;; Register configurations
(dolist (config (list
                 (my-dape-dotnet-config "ServerApp" "ServerApp" t)
                 (my-dape-dotnet-config "ClientApp" "ClientApp" t)
                 (my-dape-dotnet-config "Tests" "Tests" t)))
  (add-to-list 'dape-configs config))
#+end_src

* Part 10: Windows-Specific Setup

On Windows, Emacs needs help finding external tools since there is no standard =/usr/bin=. The approach is to keep all tools within the Emacs installation directory for portability.

** Directory Structure

#+begin_example
D:\source\emacs-30.1\
+-- bin\
|   +-- emacs.exe, runemacs.exe, etc.
|   +-- PortableGit\          # Git for version control
|   +-- csharp-ls\            # C# Language Server
|   +-- netcoredbg\           # .NET debugger
|   +-- hunspell\             # Spell checking
|   +-- find\                 # ripgrep for fast searching
|   +-- ImageMagick-...\      # Image processing
|   +-- ffmpeg-...\           # Video processing
+-- share\
    +-- emacs\...
#+end_example

** PATH Configuration

#+begin_src elisp
(when (eq system-type 'windows-nt)
  (let* ((emacs-bin (expand-file-name "~/bin"))
         (xPaths
          `(,emacs-bin
            ,(concat emacs-bin "/PortableGit/bin")
            ,(concat emacs-bin "/PortableGit/usr/bin")
            ,(concat emacs-bin "/hunspell/bin")
            ,(concat emacs-bin "/find")              ; ripgrep
            ,(concat emacs-bin "/netcoredbg")

* Part 11: Offline / Air-Gapped Environments

Setting up Emacs in an environment with no internet access requires pre-staging all components.

** Checklist

| Component                          | How to Prepare                               |
|------------------------------------+----------------------------------------------|
| Emacs source tarball               | Download =emacs-30.1.tar.gz= (~70 MB)        |
| Language server                    | Download/build and copy the binary/jar       |
| Debug adapter                      | Download and copy the binary/jar             |
| Emacs packages (dape, corfu, etc.) | Clone repos, copy =.el= files to =site-lisp= |
| Build tool caches                  | Run a full build online to populate caches   |

** Gradle Offline Builds (Java)

Run once with internet to populate the cache:
#+begin_src bash
cd /path/to/project/java
./gradlew build          # Downloads everything to ~/.gradle/
#+end_src

Then all subsequent builds work offline:
#+begin_src bash
./gradlew build --offline
#+end_src

Transfer =~/.gradle/= to the target machine. The essential directories are:

| Directory           | Size  | Purpose                    |
|---------------------+-------+----------------------------|
| =wrapper/dists/=    | ~270M | Gradle distribution binary |
| =caches/modules-2/= | ~37M  | Downloaded dependency jars |

The rest (=generated-gradle-jars=, =transforms=, =daemon/=, etc.) is regenerated on first build.

** .NET Offline Builds (C#)

For NuGet package caching:
#+begin_src bash
dotnet restore --packages ./local-packages
#+end_src

Then configure NuGet to use the local cache on the target.

* Part 12: Ada

Ada is well supported in Emacs, but the tooling landscape is fragmented. The modern =ada-mode= on MELPA (sometimes called "ada-core" or "ada-lite") depends on Alire and the Ada Language Server (ALS) for navigation and project management. In practice, getting Alire to resolve and build correctly is brittle --- it often fails with version conflicts, missing crates, or network issues, especially in offline environments. The setup is significantly more complex than other languages and frequently doesn't work.

The recommended approach is to use the *old ada-mode* (the version bundled with Emacs through Emacs 28), which provides robust Ada-aware navigation, indentation, and statement templates with no external tooling dependencies. Eglot with ALS can be layered on top when available, and debugging uses gud/gdb directly rather than dape, since there is no widely available DAP adapter for Ada.

** Choosing an Ada Mode

| Package        | Source              | Dependencies            | Navigation               |
|----------------+---------------------+-------------------------+--------------------------|
| Old ada-mode   | Emacs =<28 / manual | None                    | Built-in cross-reference |
| MELPA ada-mode | MELPA               | Alire, ALS, ada-core    | LSP via ALS              |
| ada-ts-mode    | MELPA               | Tree-sitter ada grammar | Syntax only              |

Old ada-mode is the most reliable choice. It parses Ada source directly for indentation and navigation without requiring a language server, build tool, or network access. It works immediately on any Ada project.

** Installing Old Ada-Mode

Old ada-mode was removed from Emacs 29+. To use it, copy the files from an Emacs 28 distribution or from the [[https://github.com/emacs-mirror/emacs/tree/emacs-28/lisp/progmodes][Emacs 28 source tree]]:

#+begin_src bash
mkdir -p ~/.emacs.d/old-ada-mode
# Copy these files from Emacs 28 lisp/progmodes:
#   ada-mode.el, ada-prj.el, ada-stmt.el, ada-xref.el
cp ada-mode.el ada-prj.el ada-stmt.el ada-xref.el ~/.emacs.d/old-ada-mode/
#+end_src

#+begin_src elisp
(setq my/old-ada-mode (concat user-emacs-directory "old-ada-mode"))
(when (file-exists-p my/old-ada-mode)
  (use-package ada-mode
    :load-path my/old-ada-mode))
#+end_src

** File Associations

#+begin_src elisp
(cl-loop for ext in '("\\.gpr$" "\\.ada$" "\\.ads$" "\\.adb$")
         do (add-to-list 'auto-mode-alist (cons ext 'ada-mode)))
#+end_src

** Eglot with Ada Language Server (Optional)

When ALS is available, eglot can provide LSP features on top of old ada-mode. ALS reads GNAT project files (=.gpr=) for project structure.

*** Installing ALS

ALS can be built from source or installed via Alire (when it cooperates):
#+begin_src bash
# Via Alire (if available)
alr get ada_language_server
cd ada_language_server*
alr build

# Or download a pre-built binary from:
# https://github.com/AdaCore/ada_language_server/releases
#+end_src

*** Eglot Configuration for ALS

#+begin_src elisp
(add-to-list 'eglot-server-programs
             '((ada-mode ada-ts-mode) . ("ada_language_server")))
#+end_src

To point ALS at a specific =.gpr= project file, create a =.dir-locals.el= in the project root:

#+begin_src elisp
;; .dir-locals.el
((ada-mode . ((eglot-workspace-configuration
               . ((ada . (:projectFile "my_project.gpr")))))))
#+end_src

A helper function can generate this from dired:

#+begin_src elisp
(defun my/eglot-dir-locals ()
  "Create .dir-locals.el for eglot ada-mode using the selected DIRED path."
  (interactive)
  (add-dir-local-variable
   'ada-mode
   'eglot-workspace-configuration
   `((ada . (:projectFile ,(dired-get-filename))))))
#+end_src

** Multi-Backend Navigation

Since Ada projects may use eglot, old ada-mode, or etags depending on the environment, a dispatcher function routes =M-.= and =M-,= to the correct backend automatically:

#+begin_src elisp
(defun buffer-in-eglot-mode-p ()
  (if (fboundp 'eglot-managed-p)
      (eglot-managed-p)
    nil))

(defun buffer-in-old-ada-mode-p ()
  (if (boundp 'ada-prj-default-project-file)
      t
    nil))

(defun buffer-in-tags-mode-p ()
  (if tags-table-list
      t
    nil))

(defun my/ada-find-definitions ()
  "Find definitions using the best available backend."
  (interactive)
  (cond
   ((buffer-in-eglot-mode-p)
    (message "xref: eglot")
    (xref-find-definitions (xref-backend-identifier-at-point 'eglot)))
   ((buffer-in-old-ada-mode-p)
    (message "xref: old-ada-mode")
    (setq ada-xref-other-buffer nil)
    (ada-goto-declaration (point)))
   ((buffer-in-tags-mode-p)
    (message "xref: etags")
    (my/xref-find-definitions (xref-backend-identifier-at-point 'etags)))
   (t
    (message "xref: fallback etags")
    (my/xref-find-definitions (xref-backend-identifier-at-point 'etags)))))

(defun my/ada-find-definition-pop ()
  "Pop back from definition using the matching backend."
  (interactive)
  (cond
   ((buffer-in-eglot-mode-p)
    (xref-quit-and-pop-marker-stack))
   ((buffer-in-old-ada-mode-p)
    (ada-xref-goto-previous-reference))
   (t
    (xref-go-back))))

(with-eval-after-load 'ada-mode
  (define-key ada-mode-map (kbd "M-.") 'my/ada-find-definitions)
  (define-key ada-mode-map (kbd "M-,") 'my/ada-find-definition-pop))
#+end_src

The fallback chain is: eglot (LSP) -> old ada-mode (built-in cross-reference) -> etags (static tags). This means navigation works in any environment --- from a fully connected workstation with ALS to an air-gapped machine with only the source code and a TAGS file.

** Debugging Ada with GUD/GDB

Ada programs compiled with GNAT are native binaries, so GDB supports them directly. Emacs's built-in GUD (Grand Unified Debugger) interface provides the debugging UI. There is no need for dape or a DAP adapter.

*** Compiling for Debug

#+begin_src bash
# Compile with debug symbols
gnatmake -g -O0 main.adb

# Or via a .gpr project file
gprbuild -Pmy_project.gpr -XBUILD=debug
#+end_src

*** GDB Configuration

#+begin_src elisp
(setq gdb-display-io-nopopup 1)
(setq gdb-many-windows t)         ; Show locals, stack, breakpoints, I/O

(global-set-key (kbd "<f9>")  'gud-break)
(global-set-key (kbd "<f10>") 'gud-next)
(global-set-key (kbd "<f11>") 'gud-step)
#+end_src

*** Usage

Start a debug session with =M-x gdb=, then enter the GDB command line:

#+begin_example
gdb -i=mi /path/to/my_program
#+end_example

With =gdb-many-windows= set to =t=, Emacs splits into a multi-pane layout showing source, locals, stack, breakpoints, and I/O --- similar to a traditional IDE debugger view.

| Key   | Command     | Action         |
|-------+-------------+----------------|
| =F9=  | =gud-break= | Set breakpoint |
| =F10= | =gud-next=  | Step over      |
| =F11= | =gud-step=  | Step into      |

Additional GUD commands available in the GDB buffer: =gud-cont= (continue), =gud-finish= (step out), =gud-remove= (remove breakpoint), =gud-print= (print expression).

* Troubleshooting

** Eglot won't start / "Cannot find server"

1. Check the server is on your PATH: =M-x getenv RET PATH=
2. Try running the server command manually in a shell
3. Check =*eglot-events*= buffer (=M-x eglot-events-buffer=) for detailed errors
4. For JDTLS: ensure you open a file under the project root so it finds =settings.gradle.kts= or =pom.xml=

** LSP is slow or uses too much memory

1. Disable unused server capabilities (see =eglot-ignored-server-capabilities= above)
2. Increase =read-process-output-max=
3. For JDTLS: increase =-Xmx= heap size
4. Set =corfu-auto= to =nil= and complete manually
5. For JDTLS: delete the workspace cache (=~/.cache/jdtls/=) and restart

** Debugger won't attach or launch

1. Check the =*dape-repl*= buffer for error messages
2. Verify the debug adapter runs: e.g. =netcoredbg --version= or check JDTLS bundles
3. For Java: ensure eglot is connected before starting dape
4. For .NET: ensure the project is built in Debug configuration
5. For Java attach mode: ensure the JVM was started with =--debug-jvm= or equivalent JDWP flags

** JDTLS workspace corruption

If JDTLS gets confused after project structure changes:
#+begin_src bash
rm -rf ~/.cache/jdtls/   # Or wherever -data points
#+end_src
Then =M-x eglot-shutdown= followed by =M-x eglot=.

** Wayland display issues (Linux)

If running in a VM accessed via SPICE, or if accessibility features misbehave under Wayland, switch to X11:
#+begin_src ini
# /etc/gdm/custom.conf
[daemon]
WaylandEnable=false
#+end_src

** SSH host key issues on Arch Linux

Qt =ErrorHandler::Throw= warnings from desktop SSH agents can interfere with SSH prompts. Suppress with:
#+begin_src bash
ssh -p 2222 user@host 2>/dev/null
#+end_src
